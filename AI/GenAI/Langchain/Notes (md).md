```text
Langchain:
Its a framework used to build llm apps (Conversational Chatbot, AI Assistant, AI Agents, etc).

Components:
- Models: Core interface through which we interact with llms/ models.
- Prompts: Any input/instruction or queries given to a model to guide its output.
- Chains: Wrapper that executes a defined sequence of individual components, such as models & prompts, to complete a unified task
- Indexes: Connects app to external knowledge base (Files, Websites, Databases).
- Memory: Stateful conversation with llms.
- AI Agents: LLM + Reasoning Capability + Tool Calling.

Feature:
- Concepts of Chaining
- Model Agnostic

Component of Langchain:

0) Document: Standard object used to represent a piece of text and its associated metadata. It acts as the primary data structure that flows through various components. "page_content" and "metadata".

1) Models: Core interface through which we interact with llms/ models.

 Approach:
 - LLMs: General Purpose models used for raw text generation, its traditional option. (Input = plain text , Output = plain text)
 - ChatModels: Conversational task specialized language model. (Input = msg seq, Output = chat msg)

 Feature:               LLMs                                      ChatModels
 - Memory context      No buitin memory                    Support structured conversation history
 - Role awareness      No role understanding               Understands "system", "assistant" , "user"
 - Use case            Text generation, translation   	   Conversational Ai, chatbots

2) Prompts: Any input/instruction or queries given to a model to guide its output. Single Prompt - Single turn stand alone message. Multiline Prompt - Multi-turn conversation. F-String isn't used as its doesn't provide validation.

 Approach:
 - Static Prompt - Fixed prompt template
 - Dynamic prompt - Fixed prompt template + variable placeholder (PromptTemplate, ChatPromptTemplate)

 Feature:                         PromptTemplate                                          ChatPromptTemplate
 - Usage           With standard LLMs (Text-in, Text-out).                         With Chat Models (Messages-in, message-out)
 - Structure       Single string template with placeholders (e.g., {topic}).       List of structured Messages (System, Human, AI roles).
 - Reusable        Define the logic once and inject different variables.           Entire chat schemas can be reused across different sessions.
 - Consistent      Ensures the ""instruction"" part of the prompt never changes.   Enforces consistency in how the ""System"" or ""AI"" behaves.
 - Personalized    Injects user-specific data into a flat string.                  Allows personalization through Chat History and specific roles.

Message: Data structures used to facilitate conversation between a user and an AI model.
- System Message: Used to set the behavior, persona, or ground rules for the AI.
- Human Message: Represents the prompt or query sent by the user
- AI Message: The response generated by the model. When using tool calling, this message often contains the specific instructions to invoke a tool
- Tool Message: A specific message type that sends the result of a tool's execution back to the model so it can formulate a final answer


Messages Placeholder - Injects stored chat history in prompts via PromptTemplate, ChatPromptTemplate
Note: LangChain Hub (Github for ready-to-use prompts)
______________________________________________________________________________________________________________________________________________________________________________


3) Structured Output: Enforce models to response/output in specified structure for data extraction, api building, Agents. (WithStructuredOutput - Some Models built-in feature. OutputParser - Langchain feature)

 Approach:
 - TypedDict - Dictionary of keys and value type ensure a specific structure through type hinting.
 - Complex TypedDict - Includes extras in key/value. Annotated (Adds metadata), Literal(Enum like), Optional(Specific type or None)
 - Pydantic - Data Validation and parsing python libs ensure data structured and type safe. (BaseModel, EmailStr, Field Function, Default,gt,lt,RegEx)
 - JsonSchema - Json like structure contain entire description.

 Feature:                  TypedDict                                PydanticModel                            JsonSchema
 - Type Hinting            Yes                                      Yes                                      Yes
 - Validation              No validation                            Enforce Validation                       Enforce Validation
 - Type conversion         No conversion                            Implicit conversion (coercion)           No conversion

4) Chains: An end-to-end wrapper that executes a defined sequence of individual components, such as models & prompts, to complete a unified task. Done via LCEL(Langchain Expression Language) with "|" pipe operator.

 Approach:
 - Basic/LLM Chain - Basic llm model with a prompt to execute.
 - Sequential Chain - Execute linearly where one output is input for next component via RunnableSequence/"|"
 - Parallel Chain - Execute multiple components parallel via RunnableParallel
 - Router Chain - Dynamically route inputs to different specialized sub-chains based on logic/condition via RunnableBranch.
 
get_graph, print_ascii - To print the entire chain graph

______________________________________________________________________________________________________________________________________________________________________________


5) Runnables: An standard "building blocks" of LangChain ensure that every component (Models, Prompts, Parsers) works the same way so they can be easily chained together.

 Approaches:
 - Task-Specific Runnables: Specialized components like LLMs, Prompt Template and Parsers that have been converted into Runnables to perform a single, clear action such as calling an AI model or retrieving data.
 - Runnable Primitives: Core logic blocks or orchestrators, which define the execution flow of "task specific runnables" by running them in order, simultaneously, or through conditional logic.

 Runnables Primitive Types:
 - RunnableSequence: Execute runnables sequencially one by one via "|".
 - RunnableParallel: Execute multiple runnables parallelly at once.
 - RunnableMap: Maps same input across multiple functions.
 - RunnableBranch: Implements conditional execution (if-else logic) on runnables.
 - RunnableLambda: Wraps custom python function into runnables.
 - RunnablePassthrough: Just forwards input as output (acts as placeholder).

 Feature:
 - "Pipe" Operator (|): Runnables allow you to use the | symbol to link components into a "Chain."
 - .invoke() Method: Every Runnable has a standard .invoke() command that takes an input and gives an output.
 - Consistency: Because everything is a Runnable, you can swap one model for another without changing your core logic.
 - LCEL foundation: Runnables are the engine behind LangChain Expression Language (LCEL), making complex workflows easy to read and build.

- DirectoryLoader: Load multiple files from directory(folder) of files.
- WebBaseLoader: Load ans extract text content from web-pages(URLs). (Uses BeatifulSoup to parse HTML)

7) TextSpiltters: Breaks large chunks of text/para (articles, pdfs, html pages, etc) into smaller pieces(chunks) that LLM can handle effectively.

Approaches:
- Length Based: Split based on fixed chunk size/length. (Chunk Overlap is used to maintain semantics)
- Text Structure Based: Split based on text structure in given order (paras -> lines -> words -> characters).
- Document Structure Based: Split based on documents structure (arcticle, html, code, markdown).
- Semantic Meaning Based: Splits based on semantic meaning of text using embedding models.
- ETL Tool Based: Splits based on advanced etl tools/libs (Unstructured, Docling).

CharacterTextSplitter, RecursizeCharacterTextSplitter, SemanticChunker, ChunkByTitle

8) VectorStores: System designed to store and retrieve numerical vectors (embeddings)

Features:
- Storage: Ensures that vectors and their metadata is retained in memory or on disk.
- Similarity search: Helps retrieve vectors most similar to query vector.
- Indexing: Enables fast similarity searches on high-dimensional vectors using clustering techniques.
- CRUD ops: Manages data lifecycle (add, read, update, remove vectors).

Vector Store
- Lightweight libs focus on storing vectors and similarity search.
- Don't support ACID, scaling , persistence properties like RDBMS.
- FAISS.

Vector Database
- Full-fledged database system designed to store and query vectors.
- Offer database like feature (ACID, scaling, backup, schemas, auth)
- Qdrant, Pinecone

ChromaDB is a lightweight open-source vector database.

_________________________________________________________________________________________________________________________________________________________________________


9) Retrievers: Interface to fetch relevant document objects that provide from a data source(vector store/db) to add context for a LLM response to user's query.

Approaches:
Data-source based: Varied data sources(web api, vectorstore, database)
  - Wikipedia Retriever: Queries Wikipedia api to fetch relevant content.
  - Vectorstore Retriever: Semantic search and fetch documents from vector store on embeddings.

- Search strategy based: Various retrieval techniques
  - Keyword Retriver: Fetched based on matching keywords in the documents.
  - Ensemble Retriever: Hybrid technique using Vector Retriever and BM25Retriver(Keyword Retriever).
  - Multi-Query Retriever: Multiple-query generation for same input query for Hybrid Search.

- Retrieval Quality Technique:
  - Maximum Marginal Relevance: Reduces redundancy in retrieved results while maintaining high relevance to query.
  - Reciprocal Rank Fusion: Reranking using model(Cohere) in RAG systems to combine multiple ranked lists of documents into a single, optimized ranking.
  - Contextual Compression Retriever: Improves retrieval quality by compressing documents after retrieval.
 _________________________________________________________________________________________________________________________________________________________________________

10) Tools: Python fuction/API that is packaged in way LLM can understand and call when needed.

Approaches:
- Builtin Tools: Langchain provided pre-built, production ready no setup tools.
  -DuckDuckGoSearchRun: Web search via duckduckgo
  - PythonREPLTool: Run Python raw code
  - ShellTool: Run Shell Command
  - RequestGetTool: Make HTTP Get requests
  - SQLDatabaseQueryTool: Run SQL queries

- Custom Tool:Tool that you define yourself while (calling own api, encapsulating business logic, database interaction)
  - BaseTool: Abstract base class for all tools in langchain. It defines core structure and interface that any tool must follow, whether its simple or customized function.
  - Structured Tool: Special tool type where input to tool follows a structured schema, typically defined using a Pydantic model.
  - @tool decorator: simple tool creation using decorator way.

ToolKits: Bundle of related tools that serve common purpose packaged together for convenience and reusability.
Example: GoogleDriveToolKit(GoogleDriveCreateFileTool, GoogleDriveSearchTool, GoogleDriveReadFileTool)

Tool Binding: Registering tools with LLMs for availability, tool does description, input schema via bind_tools()

Tool Calling: Process where LLM decides, during conversation or task, that it needs to use a specific tool (function) and generate a structured output (tool name, call argument)

Note: LLM doesn't actually run tool- it just suggests tool and input argument. The actual execution is handled by Langchain.

Tool Execution: Actual Python function(tool) is run using input arguments that LLM suggested during tool calling.

Feature: 
- Goal Driven
- Autonomous Planning
- Context-aware
- Reasoning-capable
- Adaptive
```

# 1. LangChain Big Picture

LangChain is useful when your LLM app needs more than a single model call.

```text
Simple LLM App:
User Query → LLM → Answer

LangChain App:
User Query → Prompt → Model → Parser → Retriever / Tool → Final Answer
```

## LangChain Layer View

| Layer | Responsibility | LangChain Component |
|---|---|---|
| Input Layer | Accept query/instruction | Prompt, Messages |
| Model Layer | Generate answer | LLMs, ChatModels |
| Workflow Layer | Connect multiple steps | Chains, Runnables |
| Knowledge Layer | External data access | Loaders, Splitters, VectorStores, Retrievers |
| Action Layer | Perform external actions | Tools, Tool Calling |
| Agent Layer | Autonomous decision-making | AI Agents, ReAct |

# 3. Models

## LLM Example



## LLM vs ChatModel

| Feature | LLM | ChatModel |
|---|---|---|
| Input | Plain string | List of messages |

| Conversation support | Weak | Strong |

# 7. Chains

A chain is useful when the workflow is fixed.

## Basic LCEL Chain

```python
from langchain_core.prompts import PromptTemplate
from langchain_openai import ChatOpenAI
from langchain_core.output_parsers import StrOutputParser

prompt = PromptTemplate.from_template(
    "Explain {topic} in simple words."
)

model = ChatOpenAI(model="gpt-4o-mini")

chain = prompt | model | parser

response = chain.invoke({"topic": "RAG"})

```

| No tool decision by itself | Can decide tools |
| Good for pipelines | Good for open-ended tasks |

# 13. Tools

# 15. ReAct Pattern

ReAct = Reasoning + Acting.

```text
Thought: I need external information.
Action: Search tool
Observation: Search result found.
Thought: Now I can answer.
Final Answer: ...
```

## ReAct Agent Example

```python
from langchain.agents import create_react_agent, AgentExecutor

agent = create_react_agent(
    llm=model,
    tools=tools,
    prompt=prompt
)

agent_executor = AgentExecutor(
    agent=agent,
    tools=tools,
    verbose=True,
    max_iterations=5
)

result = agent_executor.invoke({
    "input": "Find latest AI news and summarize it."
})
```

## Agent Loop Control

| Technique | Purpose |
|---|---|
| max_iterations | Hard stop after N steps |
| timeout | Stop after time limit |
| stop condition | Custom finish logic |
| reflection | Self-check answer quality |
| supervisor | External controller/orchestrator |

# 16. End-to-End Mini RAG Example

```python
from langchain_community.document_loaders import PyPDFLoader
from langchain_text_splitters import RecursiveCharacterTextSplitter
from langchain_community.vectorstores import FAISS
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

# 1. Load PDF
loader = PyPDFLoader("notes.pdf")
docs = loader.load()

# 2. Split documents
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50
)
chunks = splitter.split_documents(docs)

# 3. Create embeddings
embedding_model = OpenAIEmbeddings()

# 4. Store chunks in vector store
vectorstore = FAISS.from_documents(
    chunks,
    embedding_model
)

# 5. Create retriever
retriever = vectorstore.as_retriever()

# 6. Create prompt
prompt = ChatPromptTemplate.from_template("""
Answer the question using only the context below.

Context:
{context}

Question:
{question}
""")

# 7. Create model and parser
model = ChatOpenAI(model="gpt-4o-mini")
parser = StrOutputParser()

# 8. Retrieve relevant docs
question = "What is LangChain?"
retrieved_docs = retriever.invoke(question)
context = "

".join([doc.page_content for doc in retrieved_docs])

# 9. Generate final answer
chain = prompt | model | parser

answer = chain.invoke({
    "context": context,
    "question": question
})

print(answer)
```

# 19. Interview Style Q&A

## Q1. What is LangChain?

LangChain is a framework used to build LLM applications by connecting models, prompts, tools, retrievers, memory, and agents.

## Q2. What is a Runnable?

A Runnable is a standard executable unit in LangChain. Models, prompts, parsers, retrievers, and chains can behave like Runnables.

## Q3. What is LCEL?

LCEL stands for LangChain Expression Language. It allows chaining components using the `|` pipe operator.

## Q4. What is the difference between Chain and Agent?

A Chain follows a fixed flow. An Agent dynamically decides the next step using reasoning and tools.

## Q5. What is Tool Calling?

Tool Calling is when the LLM decides that a tool should be used and returns the tool name with arguments.

## Q6. Does the LLM execute tools?

No. The LLM only suggests the tool call. LangChain or your application executes the actual tool.

## Q7. What is RAG?

RAG stands for Retrieval-Augmented Generation. It retrieves external knowledge and gives it to the LLM as context.

## Q8. Why do we need Text Splitters?

Because large documents cannot be passed directly to LLMs or embedding models. Splitting improves retrieval and context quality.

## Q9. What is a Vector Store?

A Vector Store stores embeddings and allows similarity search.

## Q10. What is ReAct?

ReAct is a pattern where the agent alternates between reasoning and acting.
