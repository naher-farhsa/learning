MVC Model View Controller trio of Spring Web:
- Model -> App data like java entity class, dto 
- View -> UI page
- Controller -> request handler pass to service/business logic returns view

- Service -> part of model to actually interact with database 

Java Web: 
- HTTP -> Web Network Layer Protocol
- Servlet -> Java Class that runs on a servlet container (Tomcat, Jetty), handling HTTP req res.
  
Task of servlet: 
- Routing
- Binding
- Validation
- Exception Handling
- View resolution

- Servlet Mapping    -> URL pattern mapped to dispatcher servlet
- Dispatcher Servlet -> Front Controller that initiate routes every HTTP request to appropriate controller 
- View Resolver      -> Translate view name into real view file (jsp)
- ModelObject        -> POJO of a Class
- Model/ModelAndView -> Carry data of ModelObject from Controller to View
- ModelAttribute     -> Binds request params data to ModelObject View to Controller

Client (Form/Request)  ->  @ModelAttribute ->  POJO
POJO (processed)       ->  Model / ModelAndView ->  View (JSP/Thymeleaf)

Request Flow: 

Client-> Dispatcher Servlet -> HandlerMapping (find @Controller method)-> HandlerAdapter(call method) -> Controller -> Service -> Repository -> Controller (returns view/ JSON) -> ViewResolver (res/view/ .html) or HttpMessageConverter (JSON)-> Dispatcher Servlet -> Response

Spring AOP:
- Divide cross cutting concerns from business logic (Logging, Exception Handling, Validation, Security, Transaction)

- Aspect    -> Class that contains cross cutting logic          -- (Advice + Join Point)
- Pointcut  -> Path expression to select Join point             -- (WHERE to apply the aspect)
- Advice    -> Moment of action taken by an Aspect              -- (How and What to apply the aspect) (@Before, @After, @Around ,...)
- JoinPoint -> Specific point in method execution (method call) -- (When to apply the aspect)
- Weaving   -> Applying aspect to target
- Proxy     -> Object created by Spring to apply AOP
