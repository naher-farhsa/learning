
•Java compile and execution flow:
 source code(.java) + javac -> byte code(.class) + jvm -> machine code(.exe)

•JDK:
1. **javac**: The Java compiler, which converts source code into bytecode.
2. **java**: The launcher for running Java applications.
3. **javadoc**: Generates documentation from Java source code.
4. **jdb**: The Java debugger for debugging applications.
5. **jar**: For creating and managing JAR (Java ARchive) files.
6. **javap**: The class file disassembler for analyzing bytecode. 

•JRE:

1. **JVM (Java Virtual Machine)**: Executes Java bytecode and provides the runtime environment.
2. **Core Libraries**: Essential Java class libraries like `java.lang`,        `java.util`, and `java.io` that provide basic functionality.
3. **Class Loader**: Loads Java class files into memory as needed.
4. **Garbage Collector**: Manages memory by automatically cleaning up unused objects.

•JVM: Java bytecode (.class) to machine code (.exe) using JIT compilation

•Range and default value:
1. **byte**: 
   - Range: \(-2^{7}\) to \(2^{7} - 1\) (or -128 to 127)
   - default: 0
2. **short**: 
   - Range: \(-2^{15}\) to \(2^{15} - 1\) (or -32,768 to 32,767)
   - default: 0
3. **int**: 
   - Range: \(-2^{31}\) to \(2^{31} - 1\) (or -2,147,483,648 to 2,147,483,647)
   - default: 0

4. **char**: 
   - Range: \(0\) to \(2^{16} - 1\) (or 0 to 65,535), representing Unicode char.
   - default: '\u0000' or null char

5. **boolean**: 
   - No specific range, but it can hold one of two values: `true` or `false`.
   - default: false

6. **Object**:
   - default: null

class B {
    int a = 10; // Instance variable in class B
    void display() {
        System.out.println("Display from B");
    }
}

class C extends B {
    int a = 20; // Instance variable in class C (hides variable in B)
    void display() {
        System.out.println("Display from C");
    }
}

public class Main {
    public static void main(String[] args) {
        B b = new C();  // Upcasting to B  <============================
        b.display();    // Calls the overridden method in C
        System.out.println(b.a); // Accesses variable 'a' from B, not C

        C c = new C(); // Normal instantiation of C <=============================
        c.display();    // Calls the display method in C
        System.out.println(c.a); // Accesses variable 'a' from C
    }
}
```

### Output:

Display from C
10
Display from C
20

1. **Method Overriding**:
   - When you call `b.display()`, it executes the `display()` method from class `C`, because the method is overridden. The runtime type of `b` is `C`, so the overridden method in `C` is executed.

2. **Variable Hiding**:
   - When you access `b.a`, it refers to the variable `a` defined in class `B` (which is `10`), not `C`'s `a` (which is `20`). This is because instance    variables are not polymorphic; the reference type determines which variable is accessed.

   - When you access `c.a`, it correctly refers to `C`'s `a` (which is `20`).

Summary:
- Method calls are resolved at runtime based on the actual object type (dynamic dispatch), leading to the output from `C`.
- Variable access is resolved at compile time based on the reference type, which is why `b.a` accesses `B`'s `a`.


•Static Binding: In static binding, method or variable resolution is determined at compile time (Variable type ,overloading).

Dynamic Binding: In dynamic binding, method resolution occurs at runtime based on the actual object type. overridding, object type).
