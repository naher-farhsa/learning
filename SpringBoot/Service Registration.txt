Circuit Breaker : Acts on request-response errors/exception (api-call failure)
-----------------------------------------------------------------------------

Api-Gateway ---> Service (Down) -----(Service Specific Circuit Breaker Works)

Service ---> Service (Down) -----(Client Service Specific Feign Fallback Works)
----------------------------------------------------------------------------

Closed ----(failures exceed threshold)---> Open

CLOSED
  |
  | (failure rate exceeds threshold        (eg: For 10 total calls > 5 fail calls mean open)
  |  within sliding window calls)
  v
OPEN
  |
  | (wait-duration-in-open-state expires)
  v
HALF_OPEN
  |   (permitted calls in half open state)   (eg: 3 calls permitted in half open state)
  |
  |-- (calls succeed) ----> CLOSED
  |
  └-- (calls fail) ------> OPEN

- Eureka Server Running
- Config Server Running

2) Local bootstrap prop resolve
3) Config server connection
4) Fetch service-x (app prop) from config repo
5) Local app prop resolve        
6) Service-X spring context initialization
7) Eureka Client Initialization & Service-X Registration to Server

2) Config (Service prop)

Public Endpoint Flow: 

Assumption: 
Gateway Security Enabled
Gateway Secret Prop : Used only when feign calls are made
Gateway Auth Prop: Used only when private endpoints 
WebClient Config : Working in behind (load balancing by api-gateway for service-x)
CorsConfig : Working in behind for cross originated requests 
 
1) Client Sends Request
2) Api Gateway Receives request
3) Route Pattern Matching
4) Logging Filter (applyAuthServiceFilters) resolution
5) Forward req to auth-service
6) Auth-service feigns calls User-service (GatewaySecretProp in header)
7) Response back to client accordingly

---------------------------------------------------------------------------------------

Private Endpoint Flow:

Asumption: 
Gateway Security Enabled
Gateway Secret Prop : Used for adding request  header secret either when api-gateway calls/routes to service-x or  feign calls b/w services are made
Gateway Auth Prop: Used for token validation endpoint resolution only when private endpoints hit
WebClient Config : Working in behind (enabling req load balancing by api-gateway/auth-service for service-x)
CorsConfig : Working in behind for cross originated requests 

NOTE : applyAuthServiceFilters(), applyStandardFilters(), addGatewaySecretHeader

1) Client Sends Request
2) Api Gateway Receives request
3) Route Pattern Matching
4) Log + Auth filter (applyStandardFilters) resolution
5) Auth filter resolves Auth prop (url + validate endpoint)
6) Api-gateway forward req to auth-service (token validation)
7) Response back to api-gateway accordingly
8) Add gateway-secrets in request header (To ensure gateway approved req sent to service)
9) Finally api-gateway forward req to appropriate service

Interservice Calls via Feign

Assumptiom:
All feign calls are sent directly to

- Service-X SecretInterceptor adds 
- Service-X FeignClient use handler endpoint to call repective Service-Y
- Service-Y will reponse in accordance 
- If Service-Y down then Service-X Fallback or if Service-Y return error response then feign error decoder will map to appropriate Service-X error response
